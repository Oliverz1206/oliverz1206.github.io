## 1. SSH 是什么

SSH (Secure Shell) 是一种用于**远程登录、远程执行命令、文件传输与端口转发**的协议。日常开发里最常见的用途有三个：

- 本地终端登录远程 Linux 服务器
- 在 VSCode / Cursor 中通过 Remote SSH 打开远程目录
- 结合 `scp` / `sftp` / Git 进行文件与代码同步

对于 VSCode Remote - SSH 这类工具，本质要求是：**远端机器必须有可用的 SSH Server（通常是 `sshd`）并且网络上能通过 TCP 连接到对应端口，默认是 22。** 

最直接的 ssh 连接方式是：

```bash
ssh username@ip-address
```

输入登陆用户的密码后即可连接成功。

## 2. 生成 SSH Key 并实现免密码登录

### 2.1 基本原理

> 进行免密码登陆操作需要保证可以通过用户名密码登陆。
{: .prompt-tip }

SSH 免密码登录并不是真的“不要认证”，而是把认证方式从“输入账户密码”换成“使用一对密钥”。

- **私钥**：保存在你的本地电脑，不能泄露
- **公钥**：放到远端服务器上

连接时，服务器用公钥验证你是否持有对应私钥；验证通过后即可登录。Windows 的 OpenSSH 文档同样把这种方式称为 **key-based authentication**。 

### 2.2 推荐的 key 类型

现在一般优先使用：

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`-C` 是 comments，可以不写。如果远端环境特别老，不支持 `ed25519`，再考虑使用 RSA：

```bash
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
```

### 2.3 生成 key 的详细流程

#### a. 执行生成命令

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

执行后通常会看到：

```text
Generating public/private ed25519 key pair.
Enter file in which to save the key (/Users/yourname/.ssh/id_ed25519):
```

这里有两种常见选择：

- 直接回车：保存为默认路径 `~/.ssh/id_ed25519`
- 手动输入文件名：例如 `~/.ssh/id_ed25519_server1`

通常情况下没有必要输入新文件名，但是如果为了安全性考虑，例如公司和个人家庭网络进行隔离，
防止私钥泄露的时候不会导致影响到其他的连接，可以存储相互独立的私钥。

然后会继续提示：

```text
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
```

#### b. 是否设置 passphrase

- **设置 passphrase**：更安全。即使私钥文件泄露，别人也不能直接使用
- **不设置 passphrase**：更省事，但安全性更差

如果你经常开发，比较推荐：

- 给私钥设置 passphrase
- 再配合 `ssh-agent` 记住口令

#### c. 生成完成后的典型输出

```text
Your identification has been saved in /Users/yourname/.ssh/id_ed25519
Your public key has been saved in /Users/yourname/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx your_email@example.com
```

此时会得到两个文件：

- `id_ed25519`：私钥
- `id_ed25519.pub`：公钥

### 2.4 把公钥放到远端服务器

最常见的目标文件是远端用户目录下的：

```bash
~/.ssh/authorized_keys
```

也就是说，你需要把**本地 `.pub` 文件的内容**追加到远端的 `authorized_keys` 里。

#### a. 方法一：手动复制

先查看本地公钥：

```bash
cat ~/.ssh/id_ed25519.pub
```

把输出整行复制，然后登录到服务器，在远端执行：

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
nano ~/.ssh/authorized_keys
```

把公钥粘贴进去后保存，再执行：

```bash
chmod 600 ~/.ssh/authorized_keys
```

#### b. 方法二：使用 ssh-copy-id（Linux/macOS 常见）

```bash
ssh-copy-id username@server_ip
```

如果系统没有 `ssh-copy-id`，就用手动复制即可。

### 2.5 测试是否成功

```bash
ssh username@server_ip
```

如果配置成功，通常会出现以下情况之一：

- 直接登录成功
- 只要求输入私钥 passphrase
- 不再要求输入远端账户密码

### 2.6 常见失败原因

#### a. 权限不对

远端通常要求：

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

#### b. 用错 key

如果你有多个私钥，需要显式指定：

```bash
ssh -i ~/.ssh/id_ed25519_server1 username@server_ip
```

#### c. 服务器没允许公钥认证

Ubuntu 的 OpenSSH Server 文档说明 `sshd` 的主配置文件是 `/etc/ssh/sshd_config`，并且 `PubkeyAuthentication` 需要允许；在 Ubuntu 默认配置中该选项即便被注释，默认值通常也是 `yes`。 citeturn947524search2turn947524search11

可以检查：

```bash
sudo nano /etc/ssh/sshd_config
```

确认至少不要把下面这些关掉：

```text
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
```

修改后重启 SSH 服务。

> 如果你只是为了开发方便，并不建议关闭密码登录作为第一步。更稳妥的做法是：先把 key 登录跑通，再视情况考虑是否关闭密码认证。

## 3. VSCode / Cursor 连接问题综合

**不管你是纯终端 SSH，还是在 VSCode / Cursor 里 Remote SSH，本质上都依赖远端 SSH Server 正常监听 TCP 端口。默认就是 TCP 22。** 

所以如果出现这些问题：

- 能 ping 通，但 `ssh` 连不上
- VSCode / Cursor 能识别主机，但打不开文件夹
- Remote SSH 连上后卡住，不能浏览目录
- 不能下载文件

优先检查下面四类问题：

1. `sshd` 服务是否启动
2. TCP 22 端口是否在监听
3. 防火墙是否放行 22 端口
4. 认证方式是否正常（密码 / 公钥）

### 3.1 Windows

#### a. 打开 SSH 服务

Microsoft 官方文档说明，Windows 支持安装并启用 OpenSSH Server；启用后需要启动 **OpenSSH SSH Server** 服务，也就是 `sshd` 服务。可以在 `services.msc` 中找到 `OpenSSH SSH Server` 服务，并把启动方式设为自动。 

或者直接在 powershell 中：

```powershell
Get-Service sshd
Start-Service sshd
Set-Service -Name sshd -StartupType Automatic
```

#### b. 检查 22 端口是否监听

```powershell
netstat -ano | findstr :22
```

如果看到 `LISTENING`，说明 TCP 22 已经在监听。

#### c. 防火墙

OpenSSH Server 的官方安装/启用文档明确提到需要配置防火墙放行 SSH。 citeturn947524search0

可以检查 Windows Defender Firewall 是否允许 22 端口或 OpenSSH Server。

### 3.2 Linux 

#### a. 打开 SSH 服务

Ubuntu 官方文档说明，安装 OpenSSH Server 的包是：

```bash
sudo apt install openssh-server
```

安装后主要配置文件位于 `/etc/ssh/sshd_config`，服务名是 `sshd`。

#### b. 启动并设置开机自启

```bash
sudo systemctl status ssh
sudo systemctl enable ssh
sudo systemctl start ssh
```

有些发行版服务名显示为 `ssh`，有些底层进程名是 `sshd`，本质上都是 OpenSSH Server。Ubuntu 文档强调配置对象就是 `sshd`，配置文件是 `/etc/ssh/sshd_config`。 citeturn947524search2

#### c. 检查端口监听

```bash
ss -tlnp | grep :22
```

或者：

```bash
sudo netstat -tlnp | grep :22
```

#### d. UFW 防火墙放行

如果系统启用了 UFW：

```bash
sudo ufw allow 22/tcp
sudo ufw status
```

### 3.3 VSCode / Cursor 里 “连得上但打不开文件夹” 的常见原因

#### a. ssh 其实没真正连通

先不要直接怀疑 VSCode / Cursor，先在本地终端验证：

```bash
ssh username@server_ip
```

只有终端 SSH 先稳定可用，Remote SSH 才更容易正常。

#### b. 远端目录权限不对

例如你连接的是普通用户，但尝试打开的是无权限目录。

#### c. 远端 shell 初始化脚本有问题

例如 `.bashrc`、`.zshrc` 里输出了很多额外内容，可能影响 Remote SSH 初始化。

#### d. 远端磁盘满了 / home 目录异常

VSCode Remote SSH 需要在远端用户目录里部署一些服务端组件；如果磁盘满了，也会出现很怪的问题。VS Code 官方文档说明 Remote SSH 连接后会在远端机器上工作，并与远端文件系统交互。 

> 排查顺序建议永远是：**先终端 SSH，后 VSCode/Cursor；先服务，后插件；先网络，后编辑器。**
{: .prompt-info }

## 4. VSCode / Cursor 中的 SSH Config 写法

VS Code 官方文档说明 Remote - SSH 使用标准的 SSH 配置文件；常见写法就是在 `~/.ssh/config` 中定义主机别名。Cursor 在这一点上通常也是沿用同样的 OpenSSH 配置语法。 

### 4.1 SSH config 文件位置

常见位置：

```text
~/.ssh/config
```

Windows 上通常是：

```text
C:\Users\你的用户名\.ssh\config
```

如果文件不存在，可以手动创建。

### 4.2 最基础的示例

```sshconfig
Host myserver
    HostName 192.168.1.100
    User yunfan
    Port 22
    IdentityFile ~/.ssh/id_ed25519
```

之后你就可以直接：

```bash
ssh myserver
```

VSCode / Cursor 里也通常直接选择 `myserver` 即可。

### 4.3 最常用的 keywords

下面这些是日常最常用的：

| Keyword                 | 作用                 | 示例                                    |
| ----------------------- | -------------------- | --------------------------------------- |
| `Host`                  | 主机别名             | `Host gpu-server`                       |
| `HostName`              | 真正的 IP 或域名     | `HostName 10.0.0.8`                     |
| `User`                  | 登录用户名           | `User ubuntu`                           |
| `Port`                  | SSH 端口             | `Port 22`                               |
| `IdentityFile`          | 指定私钥路径         | `IdentityFile ~/.ssh/id_ed25519`        |
| `IdentitiesOnly`        | 只使用显式指定的 key | `IdentitiesOnly yes`                    |
| `ForwardAgent`          | 转发本地 ssh-agent   | `ForwardAgent yes`                      |
| `ServerAliveInterval`   | 定时保活秒数         | `ServerAliveInterval 60`                |
| `ServerAliveCountMax`   | 保活失败次数阈值     | `ServerAliveCountMax 3`                 |
| `ProxyJump`             | 跳板机               | `ProxyJump bastion`                     |
| `ProxyCommand`          | 自定义代理命令       | `ProxyCommand ...`                      |
| `LocalForward`          | 本地端口转发         | `LocalForward 8888 127.0.0.1:8888`      |
| `RemoteForward`         | 远端端口转发         | `RemoteForward 8080 127.0.0.1:8080`     |
| `StrictHostKeyChecking` | 主机指纹检查策略     | `StrictHostKeyChecking ask`             |
| `UserKnownHostsFile`    | known_hosts 文件位置 | `UserKnownHostsFile ~/.ssh/known_hosts` |

### 4.4 使用跳板机的示例

```bash
Host bastion
    HostName 1.2.3.4
    User jumpuser
    IdentityFile ~/.ssh/id_ed25519_jump

Host internal-gpu
    HostName 10.10.0.25
    User gpuuser
    IdentityFile ~/.ssh/id_ed25519_gpu
    ProxyJump bastion
```

此时执行：

```bash
ssh internal-gpu
```

会先经过 `bastion` 再进入内网主机。

### 4.5 SSH 端口转发：通过跳板机访问内网主机的 VNC

有些时候目标主机并不直接暴露在当前网络中，例如它只在 VPN 或内网里可访问。这时可以使用 SSH 的**本地端口转发（Local Port Forwarding）**，把远端主机上的某个端口映射到自己电脑的本地端口，再让本地软件去连接这个本地端口。

例如下面这个场景中，我想通过 VNC 访问一台只能经由跳板机到达的目标主机：

```bash
Host nbser_vnc.xxx.tech
    HostName 10.1.1.102
    User yfzhang
    ProxyCommand ssh -W %h:%p 10.1.0.8
    IdentityFile ~/.ssh/id_rsa
    LocalForward 10022 127.0.0.1:5800
```

其含义如下：
- Host nbser_vnc.xxx.tech：给这组 SSH 配置起一个别名，后续可以直接使用 ssh nbser_vnc.xxx.tech
- HostName 10.1.1.102：真正要连接的目标主机
- User yfzhang：登录该主机时使用的用户名
- ProxyCommand ssh -W %h:%p 10.1.0.8：先 SSH 到跳板机 10.1.0.8，再由跳板机继续转发到目标主机
- IdentityFile ~/.ssh/id_rsa：指定使用的私钥文件
- LocalForward 10022 127.0.0.1:5800：把本机的 10022 端口转发到目标主机自己的 127.0.0.1:5800

例如建立连接后，可以这样使用：

```bash
$ ssh nbser_vnc.xxx.tech
```

然后在 VNC 客户端里填：

```bash
127.0.0.1:5800
```

即可连接显示。

如果不想写进 ~/.ssh/config，也可以直接在命令行中临时写成一条：

```bash
$ ssh -N -L 10022:127.0.0.1:5800 -J 10.1.0.8 yfzhang@10.1.1.102
```

其中：
- `-N`：不执行远程命令，只建立隧道
- `-L`：本地端口转发
- `-J`：通过跳板机进行连接（等价于某些 ProxyCommand 的写法）

对于 VNC 这类服务，目标端经常只监听在 127.0.0.1，也就是“只允许本机访问”。这时 SSH 转发就尤其有用，因为它相当于把远端的本地服务安全地“带”到了自己电脑上。

## 5. 总结

SSH 相关问题其实可以统一成三层：

1. **密钥层**：有没有生成 key，公钥有没有放对，权限对不对
2. **服务层**：`sshd` 有没有启动，TCP 22 有没有监听，防火墙有没有放行
3. **工具层**：VSCode / Cursor 的 SSH config 是否写对，是否只是上层 Remote SSH 初始化异常

只要你按这个顺序排查，绝大多数 SSH 问题都不会太乱。
